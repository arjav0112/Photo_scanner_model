# HOW THE SEARCH FUNCTION WORKS - STEP BY STEP

## CURRENT WORKFLOW (What happens when you search)

### Step 1: User Input
```
Command: python main.py search "Image of mountains and valley"
```

### Step 2: Load AI Model (main.py, line 32-34)
```python
handler = MobileCLIPHandler()
```
- Loads the CLIP model that can understand both images and text
- This model converts text and images into 512-dimensional vectors

### Step 3: Convert Query to Embedding (main.py, line 36-40)
```python
text_emb = handler.get_text_embedding(args.query)
text_emb = text_emb / np.linalg.norm(text_emb)  # Normalize
```
- Your search query "Image of mountains and valley" is converted to a 512-dimensional vector
- This vector represents the semantic meaning of your query
- Normalization ensures the vector has length 1 (for proper comparison)

### Step 4: Load Database (main.py, line 42-48)
```python
db = PhotoDatabase()
batch_gen = db.get_search_data_generator(batch_size=2048)
```
- Opens photos.db database
- Creates a generator to load images in batches (memory efficient)
- **CURRENTLY RETRIEVES: paths, embeddings, ocr_text**
- **NOT RETRIEVED: metadata** ❌

### Step 5: Process Each Batch (main.py, line 55-74)
```python
for paths, image_embs, ocr_texts in batch_gen:
```

For each batch of images:

#### 5a. Normalize Image Embeddings (line 59-61)
```python
norms = np.linalg.norm(image_embs, axis=1, keepdims=True)
image_embs = image_embs / (norms + 1e-8)
```
- Normalize all image vectors to length 1

#### 5b. Calculate Vector Similarity Scores (line 63-64)
```python
vector_scores = np.dot(image_embs, text_emb)
```
- **This is the MAIN scoring mechanism**
- Computes cosine similarity between query and each image
- Formula: similarity = query_vector · image_vector
- Range: -1 to 1 (higher = more similar)
- Example: If query is "cat" and image is a cat, score might be 0.85

#### 5c. Calculate OCR Text Scores (line 66-71)
```python
ocr_scores = np.zeros(count)
if has_text_query:
    for i, text in enumerate(ocr_texts):
        if text and query_lower in text.lower():
            ocr_scores[i] = 1.0
```
- **Simple text matching** (exact substring match)
- If your query appears in the OCR text, add +1.0 to score
- Example: Query "Pankhni" would match image with OCR "Pankhni Ethnic Wear"

#### 5d. Combine Scores (line 73-74)
```python
final_batch_scores = vector_scores + ocr_scores
```
- **Final score = Vector similarity + OCR bonus**
- Example: 
  - Image of cat with no text: 0.85 + 0 = 0.85
  - Image with matching text: 0.30 + 1.0 = 1.30

### Step 6: Collect All Results (line 78-84)
```python
all_results.append({
    "path": paths[i],
    "score": final_batch_scores[i],
    "v_score": vector_scores[i],
    "o_score": ocr_scores[i]
})
```
- Store all images with their scores

### Step 7: Apply Adaptive Filtering (line 92-140)
```python
all_results.sort(key=lambda x: x["score"], reverse=True)
score_threshold = max(top_score * (1 - RELATIVE_THRESHOLD), MIN_ABSOLUTE_SCORE)
top_results = [res for res in all_results if res["score"] >= score_threshold]
```
- Sort by score (highest first)
- Calculate dynamic threshold
- Filter out low-scoring images

### Step 8: Display Results (line 145-153)
```python
for res in top_results:
    print(f"Score: {res['score']:.4f} | File: {res['path']}")
```

Looking at the database, we have rich metadata that is **STORED but IGNORED**:

### Available Metadata (from inspect_db.py output):

1. **Device Information**
   - Example: "OnePlus OnePlus 11R 5G"
   - Could match queries like "photos from OnePlus" or "phone photos"

2. **Date/Time Taken**
   - Example: "2026:01:10 13:20:34"
   - Could match queries like "photos from January" or "recent photos"

3. **GPS Location**
   - Coordinates: 28.708586, 77.106336
   - Location name: "North West Delhi, Pitampura, NCT, India"
   - Could match queries like "photos from Delhi" or "India photos"

4. **Camera Settings**
   - ISO, aperture, shutter speed, focal length
   - Example: "ISO 640 | f/1.8 | 1/50 | 5.6mm"
   - Could match queries like "low light photos" or "portrait mode"

5. **Altitude**
   - Example: "182.4m"
   - Could match queries like "mountain photos" (high altitude)

6. **Image Dimensions**
   - Example: "3072x4096"
   - Could match queries like "high resolution" or "vertical photos"

## THE PROBLEM

### Current Search Flow:
```
Query → Text Embedding → Compare with Image Embeddings → OCR Match → Score
                                                                        ↓
                                                            Metadata IGNORED ❌
```

### What's Missing:
```
Database Query (line 147):
SELECT path, embedding, ocr_text FROM photos
                                    ↑
                            metadata column NOT retrieved!
```

The `get_search_data_generator()` function only retrieves:
- path
- embedding  
- ocr_text

It does **NOT** retrieve the `metadata` column!

---

## EXAMPLE: Why This Matters

### Query: "photos from Delhi"

**Current behavior:**
1. Convert "photos from Delhi" to embedding
2. Compare with image embeddings (visual similarity)
3. Check OCR text for "Delhi"
4. Result: Low scores because images don't visually look like "Delhi"

**Expected behavior:**
1. All of the above, PLUS
2. Check metadata for location = "Delhi"
3. Boost score for images taken in Delhi
4. Result: High scores for Delhi photos!

### Query: "photos taken in January 2026"

**Current behavior:**
- Only visual similarity (meaningless)
- Result: Random images

**Expected behavior:**
- Parse date from metadata
- Match images from January 2026
- Result: Correct time-based filtering!

## TECHNICAL DETAILS

### Database Schema:
```sql
CREATE TABLE photos (
    id INTEGER PRIMARY KEY,
    path TEXT,
    filename TEXT,
    size_bytes INTEGER,
    modified_time REAL,
    embedding BLOB,          -- ✅ Used for vector search
    metadata TEXT,           -- ❌ NOT used in search!
    ocr_text TEXT            -- ✅ Used for text matching
)
```

### Current Data Flow:
```
Scanner → Extract metadata → Store in DB → Search IGNORES it
```

This is why your search for "mountains and valley" only got score 0.2542 - 
it's purely based on visual similarity, not considering that the image 
metadata might say "Location: Himalayan Valley" or "Date: Mountain Trek 2026"!
